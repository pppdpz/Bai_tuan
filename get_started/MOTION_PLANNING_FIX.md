# 运动规划问题分析与解决方案

## 问题诊断

### 症状
机械臂在Step 0能够接近小卫星太阳能板，但之后迅速远离，没有执行抓取动作。

### 根本原因

从日志分析发现：

```
Step 50 (阶段2-精确定位):
  目标 TCP: [0.15, 0.24, 1.43]
  实际 TCP: [0.0796, 0.1410, 1.4319]
  偏差: X=0.07m, Y=0.10m

Step 70 (阶段3-抓取):
  目标 TCP: [0.15, 0.24, 1.28]
  实际 TCP: [0.2295, -0.5983, 1.1370]
  偏差: X=0.08m, Y=0.84m ⚠️
```

**核心问题**：原代码只在阶段切换点（step 0, 50, 70...）求解IK，然后在整个阶段内保持该目标不变。这导致：

1. IK求解器给出正确的关节角度
2. 但机械臂需要时间才能到达目标位置
3. 在还没到达时，就切换到下一个目标
4. 结果是机械臂永远在"追赶"目标，但永远追不上

这就像开车时，刚踩油门准备去A点，还没到就被告知要去B点，导致永远在路上。

## 解决方案

### 方案1：每步求解IK（已实施）✅

**原理**：每个仿真步都重新求解IK，实现真正的笛卡尔空间轨迹跟踪。

**优点**：
- 机械臂会精确跟随笛卡尔空间的轨迹
- 即使有外力干扰，也能自动纠正
- 适合需要高精度路径跟踪的任务

**缺点**：
- 计算量稍大（每步都求解IK）
- 但对于220步的短任务，完全可以接受

**代码改动**：
```python
# 原代码：只在阶段切换点求解IK
if step in [0, 50, 70, 90, 170, 190]:
    q_sol, ik_success = ik_solver.solve_ik_batch(...)
    target_joint_angles = q_sol.clone()

# 新代码：每步都求解IK
q_sol, ik_success = ik_solver.solve_ik_batch(
    ee_pos_batch,
    ee_quat_batch,
    curr_q  # 使用当前关节角度作为初始值
)
target_joint_angles = q_sol.clone()
```

### 方案2：关节空间插值（备选）

如果计算资源有限，可以使用关节空间插值：

```python
if step in [0, 50, 70, 90, 170, 190]:
    # 求解目标关节角度
    q_target, _ = ik_solver.solve_ik_batch(...)
    q_start = curr_q.clone()
    stage_start_step = step

# 在阶段内部插值
t = (step - stage_start_step) / stage_duration
q_interpolated = q_start + t * (q_target - q_start)
```

**优点**：计算量小
**缺点**：路径不是笛卡尔空间的直线，可能碰撞

### 方案3：增加等待时间（不推荐）

延长每个阶段的步数，给机械臂更多时间到达目标。

**缺点**：
- 治标不治本
- 任务执行时间变长
- 仍然无法保证精确到达

## 技术细节

### IK求解器配置
- 使用 pyroki 求解器
- 每步使用当前关节角度作为seed（初始猜测）
- 这确保了解的连续性，避免突然跳跃

### 坐标系转换
- TCP (pinch点) 相对 wrist_3_link 的偏移：
  - 位置: [0.0, 0.2558, 0.0] m
  - 姿态: [0.5, 0.5, 0.5, -0.5] (四元数)
- 使用 `tcp_pose_to_wrist3_pose()` 函数转换
- 转换精度：0.0000mm（已验证）

### 轨迹规划
- 使用S曲线插值 `smooth_interpolation(t)` 提供平滑加减速
- 6个阶段：接近 → 精确定位 → 抓取 → 展开 → 释放 → 撤离
- 总步数：220步

## 下一步优化

如果仍有问题，可以考虑：

1. **调整PID参数**：如果机械臂响应太慢
2. **增加阶段步数**：如果运动太快导致不稳定
3. **添加碰撞检测**：避免与卫星本体碰撞
4. **优化抓取点**：如果夹爪无法稳定抓取
